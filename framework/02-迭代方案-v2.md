# OntoDerive v2 设计方案——多理论融合的迭代架构

> 基于贝叶斯理论·逻辑论·信息论·图灵机·系统论·控制论的融合方案。
> 以本体论为内核，形式语言为外壳，构建可自演化、可量化、可证明的知识工程框架。

---

## 一、当前系统的六维诊断

### 1.1 贝叶斯视角：确定性而非概率性

| 现状 | 问题 |
|------|------|
| 置信度分4级(fact/inference/hypothesis/estimated) | 无概率量化，不能做信念传播 |
| 事实更新后手动标记needs_review | 没有自动置信度重新计算 |
| 推论之间没有条件独立性假设 | 无法进行贝叶斯更新 |

### 1.2 逻辑视角：非形式化推理

| 现状 | 问题 |
|------|------|
| derives_from 是自然语言标注 | 不是形式化的逻辑蕴含(⊨) |
| 推理规则用Python硬编码 | 没有形式语法和推理演算 |
| 规约检查是启发式grep | 不是定理证明 |

### 1.3 信息论视角：无量化度量

| 现状 | 问题 |
|------|------|
| 无信息熵度量 | 不知道知识库的不确定性 |
| 无信息增益计算 | 不知道新增事实的价值 |
| 无压缩比度量 | 不知道推导链的效率 |

### 1.4 图灵机视角：非计算完备

| 现状 | 问题 |
|------|------|
| 推导引擎是脚本，不是状态机 | 没有形式化的计算模型 |
| 无停机保证 | 多轮迭代可能不收敛 |
| 无状态持久化 | 推导过程不可重放 |

### 1.5 系统论视角：开环而非闭环

| 现状 | 问题 |
|------|------|
| 手动触发推导 | 没有自动感知-决策-执行循环 |
| 规约违规只报告 | 不自动生成修复 |
| 无学习能力 | 历史模式不沉淀 |

### 1.6 控制论视角：无反馈控制

| 现状 | 问题 |
|------|------|
| 设定期望值(规约)但不自动跟踪偏差 | 没有比例-积分-微分(PID)控制 |
| 无自适应阈值 | 规约阈值不随系统状态调整 |
| 无稳定性分析 | 不知道迭代是否收敛 |

## 二、融合架构：六层一体化设计

### 2.1 模型分层架构（MOF四层 + 本体双层）

参照OMG的MOF（Meta-Object Facility）四层架构和本体论的分层思想：

```
┌─────────────────────────────────────────────────────────────┐
│  M3: 元元模型层 (Meta-Meta-Model)                            │
│  "建模语言的定义语言" — 定义元模型的结构和语法                   │
│  形式：OntoLang BNF元语法 · 类型构造子 · 元关系矩阵           │
│  实例：Entity ::= Type { property: Type }                     │
├─────────────────────────────────────────────────────────────┤
│  M2: 元模型层 (Meta-Model)                                   │
│  "建模语言" — 定义领域模型的构建规则                            │
│  形式：8元类型 · 4元关系 · 4元约束 · 元关系矩阵               │
│  实例：DOMAIN ::= ORG | ROL | PRJ | RES                       │
├─────────────────────────────────────────────────────────────┤
│  M1: 模型层 (Model)                                          │
│  "领域模型" — 特定领域的概念化                                 │
│  形式：实体实例 · 关系实例 · 推导链 · 状态机                    │
│  实例：ORG-国转中心, INF-L1, T0→T1                            │
├─────────────────────────────────────────────────────────────┤
│  M0: 实例/数据层 (Instance/Data)                              │
│  "现实世界对象" — 具体事实和数据点                              │
│  形式：DAT-D-F1=470, POL-P-F9="京教科人办发[2026]2号"          │
│  实例：470项成果, 133名经理人, 220亿基金                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 本体双层

```
┌─────────────────────────────────────────────────────────────┐
│  元本体层 (Meta-Ontology)                                    │

│  形式：TBox(术语箱) + ABox(断言箱)                             │

```

### 2.4 向上规约·向下推导（核心机制）

**向下推导 (Top-Down Derivation)**：
```
M3 元元模型层 · 形式语言规则
  ↓ 实例化
M2 元模型层 · 8元类型约束
  ↓ 映射
M1 模型层 · 领域本体
  ↓ 实例化
M0 数据层 · 具体事实
  ↓ 归纳
方案文档 · 结构化产出
```

**向上规约 (Bottom-Up Protocol Abstraction)**：
```
M0 数据层 · 具体事实和观测
  ↑ 抽象化
M1 模型层 · 识别模式和实体
  ↑ 归纳
M2 元模型层 · 提炼规约规则
  ↑ 形式化
M3 元元模型层 · 更新形式语言
```

**闭环循环**：
```
现实世界 → [M0] 数据采集 → [M1] 本体建模 → [M2] 元模型规约 → [M3] 形式化
    ↑                                                                  │
    └────────────── [方案产出] ←─── [向下推导] ←─────────────────────────┘
```

这正是OntoDerive的"双向能力"在模型理论中的形式化表达：
- **向上规约**：从具体方案中抽象出可复用的规约规则（更新protocols/）
- **向下推导**：从元模型和规约出发，推导出具体方案内容（填充scheme/）

## 三、分层详细设计

### 3.1 M3 元元模型层：形式语言与元本体

**定位**：建模语言的"语法手册"。定义什么是合法的实体定义、关系声明、规约表达式。

**核心构件**：

```
OntoLang BNF元语法:
  EntityDef    ::= "entity" ID ":" Type "{" Property* "}"
  RelationDef  ::= Subject "→" RelationType "→" Object
  InferenceDef ::= "infer" ID ":" Type "{" Premise+ Conclusion "}"
  ProtocolDef  ::= "protocol" ID ":" ConstraintType "{" LogicExpr "}"
  
  Type         ::= "DOMAIN" | "FACT" | "INFERENCE" | "STATE" 
                 | "DOCUMENT" | "CONSTRAINT" | "PROCESSOR"
  RelationType ::= "STRUCT" | "DERIVE" | "BEHAVIOR" | "JUSTIFY"
  
元元公理(不可违反):
  Axiom-1: ∀e ∈ Entity, e.type ∈ {7种元类型}
  Axiom-2: 关系必须满足元关系矩阵的域/值域约束
  Axiom-3: 元类型不可交叉(DOMAIN∩FACT=∅)
  Axiom-4: 推导方向必须从高阶到低阶(M3→M2→M1→M0)
```

### 3.2 M2 元模型层：本体核 (升级)

**定位**：建模语言的"词汇手册"。定义8元类型、4元关系、4元约束的具体规范。

**升级点**：
1. 类型定义从自然语言表格 → 形式化类型构造子
2. 实体实例必须满足类型约束(statically checked)
3. 关系类型定义域和值域(domain/range约束)
4. 元关系矩阵形式化为一阶逻辑公理

**类型构造子**：
```text
DOMAIN(type) ::= ORG(name) | ROL(name) | PRJ(name) | RES(name)
FACT(type)   ::= DAT(id, value, source) | POL(id, authority, date)
INFERENCE(t) ::= CONTRADICTION | BUSINESS | ARCHITECTURE
STATE(type)  ::= T(index) | F(index) | H(index)
DOCUMENT(t)  ::= COL | DOC | CH | SEC
CONSTRAINT(t)::= CON(name)
PROCESSOR(t) ::= ENG(name)

-- 元关系矩阵的公理化
∀s,o: DERIVE(s,o) → s.type ∈ {FACT, INFERENCE} 
                   ∧ o.type ∈ {INFERENCE, DOCUMENT}
∀s,o: BEHAVIOR(s,o) → s.type = STATE ∧ o.type = STATE
```

### 3.3 M1 模型层：领域本体

**定位**：特定领域的概念化。即现有SSOT的实体实例。

**TBox(术语箱)**：
```text
概念: 国转中心, 技术经理人, 转化旅程
属性: 运行机制="双轨", 转化率=3.2%
角色: 高校科研人员(成果供给方), 企业用户(需求方)
```

**ABox(断言箱)**：
```text
实例: ORG-国转中心 : Organization
      ROL-技术经理人 : Role
      INF-L1 : Contradiction
断言: ORG-国转中心.running_mechanism = "双轨"
      D-F1.value = 470
```

### 3.4 M0 实例/数据层：事实基座

**定位**：原始数据和不可辩驳的事实。即现有SSOT的02-事实基座。

**数据事实**：
- D-F1=470, D-F3=15, D-F9=133, D-F13=220亿, D-F14=6.2亿
- 所有数据必须有来源+时间戳

**政策事实**：
- P-F9=京教科人办发[2026]2号
- 所有政策必须有文号+发布主体

### 3.5 L4 计算基座：知识状态机 (Turing-K)

### 3.3 L2 形式语言层：OntoLang

定义一种领域专用语言(DSL)：

```
-- 实体定义
entity ORG-国转中心 : Organization {
    governance: "理事会领导下的主任负责制"
    mechanism: "事业+企业双轨"
}

-- 事实声明
fact D-F1 : DataPoint {
    value: 470
    unit: "项"
    source: "中心介绍PDF"
}

-- 推论(逻辑蕴含)
inference INF-L1 : Contradiction {
    derives_from: [D-F1, D-F3, ORG-国转中心]
    theory: "双元性组织理论(Tushman & O'Reilly, 1996)"
    confidence: 0.92
}

-- 规约(形式约束)
protocol P-SCHEME-002 : Constraint {
    ∀a ∈ Assertion, has_keyword(a, "必须") → ∃ref ∈ SSOT_ID, contains(a, ref)
}
```

**语法形式**：BNF + 类型标注，可被静态分析，可映射到一阶逻辑。

### 3.4 L3 逻辑层：蕴含图与证明论

**蕴含图**(Entailment Graph)：将SSOT建模为有向图

```
节点 = 事实 + 推论
边   = derives_from(实质蕴含→)
权重 = 置信度

性质：
- 路径 = 推导链
- 环   = 循环论证(需检测)
- 根节点 = 事实(不可再推导)
- 叶节点 = 最终推论
- 深度 = 推导链长度
```

**证明论**：
- 每条INF推论附带一个"证明项"：`INF-L1 := [D-F1, D-F3, ORG-国转中心] ⊢ 红蓝双轨分离`
- 格式：`⊢`左边是前提，右边是结论
- 证明检查：grep前提中的实体ID是否在文本中出现
- 一致性检查：不存在两个推理链推导出矛盾的结论

**模态逻辑扩展**：
- `□P` (必然P)：P被所有事实支持，不可修正
- `◇P` (可能P)：P被部分事实支持，可被新事实修正
- `¬◇P` (不可能P)：P与已知事实矛盾
- 当前标记：fact → □, inference → ◇ (可被修正)

### 3.5 L4 贝叶斯层：概率本体

将确定性本体扩展为**概率图模型**：

```
每个实体e有一个置信度分布 P(e)
每个关系r(e1, e2)有一个条件概率 P(e2 | e1)
当新事实到达时，执行信念传播(Belief Propagation)
```

**置信度计算**(从四级标签到连续概率)：

| 当前标签 | 映射概率 | 贝叶斯解释 |
|---------|---------|-----------|
| fact | 0.95-0.99 | 多次验证，后验概率极高 |
| inference | 0.70-0.94 | 推导而来，取决于前提置信度 |
| hypothesis | 0.30-0.69 | 有初步证据但未充分验证 |
| estimated | 0.05-0.29 | 估算值，置信度低 |

**信念传播规则**：
```
P(INF | premises) = Π P(premise_i) × P(rule | premises)
P(INF updated | new_evidence) ∝ P(new_evidence | INF) × P(INF)
```

**信息熵**：
```
H(KB) = -Σ P(e) × log(P(e))  对整个知识库
IG(new_fact) = H(KB_before) - H(KB_after)  信息增益
```

### 3.6 L5 信息论层：知识质量指数

定义一组可量化的知识质量指标：

| 指标 | 公式 | 含义 |
|------|------|------|
| KQ-熵 | H(KB) = -ΣP(e)logP(e) | 总体不确定性 |
| KQ-精度 | avg P(e) for all e | 平均置信度 |
| KQ-密度 | #inferences / #facts | 推导深度 |
| KQ-覆盖 | #mapped / #total | 方案覆盖度 |
| KQ-闭环 | #resolved_gaps / #total_gaps | 缺口闭环率 |
| KQ-压缩 | |KB| / |min_axioms| | 推导效率 |

**信息增益引导**：当需要选择"下一个最有价值的事实"时，选择IG最大的方向。

### 3.7 L6 控制论层：自适应反馈系统

```
                    ┌─────────────┐
  设定点(规约) ───→│   比较器    │←─── 测量(当前状态)
                    └──────┬─────┘
                           │ 误差(规约违规)
                           ▼
                    ┌─────────────┐
                    │   PID控制器  │
                    └──────┬─────┘
                           │ 控制信号(修复建议)
                           ▼
                    ┌─────────────┐
                    │   执行器    │───→ 系统(知识库+方案)
                    │ (推导引擎)  │
                    └─────────────┘
```

**PID三个分量**：
- **P(比例)**：当前规约违规数 → 立即修复
- **I(积分)**：历史违规累积 → 系统性问题(规约本身需要调整)
- **D(微分)**：违规变化率 → 预测即将发生的违规

**自适应阈值**：
- 规约阈值不是固定的(如"断言追溯率>=70%")
- 而是根据历史数据动态调整(取P50/P80/P95分位)
- 新项目使用默认阈值，运行3轮后校准

**自演化**：
- 记录每次推导的收敛速度(从多少缺口→0缺口需要几轮)
- 如果收敛速度持续下降 → 提示规约需要简化
- 如果收敛速度稳定 → 可增加更高要求的规约

## 七、关键融合点：六论+模型理论如何协同

### 4.1 核心循环

```
[信息论] 测量知识库熵
    ↓ 熵高 → 需要更多事实
[贝叶斯] 计算信息增益最大的方向
    ↓
[控制论] 生成"最有价值事实"采集建议
    ↓ 新事实到达
[图灵机] 执行状态转移(推导)
    ↓
[逻辑] 检查推导的一致性和证明链
    ↓ 发现违规
[控制论] 生成修复信号
    ↓ 修复完成，回到[信息论]重新度量
```

### 4.2 形式语言的统一表达

所有六个理论的概念用**同一种形式语言**(OntoLang)表达：

```text
-- 贝叶斯：条件概率作为关系属性
relation R-INF-001 (premises: Entity[], conclusion: Inference) {
    conditional_probability: 0.85
    evidence_strength: "strong"
}

-- 逻辑：蕴含作为基本操作
axiom implication_chain {
    (A → B) ∧ (B → C) → (A → C)  // 传递性
}

-- 信息论：熵作为元属性
meta KB_entropy = -Σ P(e) * log(P(e))

-- 控制论：反馈回路作为系统结构
feedback_loop protocol_check {
    measure: check_result.passed_ratio
    setpoint: 0.80  // 80%通过率
    action: generate_fixes(check_result)
}
```

### 4.3 一个具体例子

以"断言追溯率低"为例，展示各论如何协同：

```
1. [控制论] 设定点=70%追溯率，当前测量=12.9% → 误差=57.1%
2. [信息论] 计算裸断言的信息熵 → 确认问题的严重程度
3. [逻辑] 定位裸断言(116个中断言中101个无编号)
4. [贝叶斯] 计算"添加编号"的预期信息增益 → 确定优先级
5. [图灵机] 执行状态转移：为断言添加编号
6. [控制论] 重新测量 → 如果误差缩小 → 继续，否则调整策略
7. [形式语言] 所有操作记录为OntoLang表达式，可回放
```

## 六、向上规约·向下推导：闭环机制详解

### 6.1 向下推导的实现

从M3到M0的逐层实例化：

```
输入: M3 (元元模型约束) + M2 (元模型规则) + M1 (领域需求)
输出: M0 (具体事实采集指令) + 方案文档

步骤:
1. M3 → M2: OntoLang编译器读取BNF元语法
   → 生成实体模板和关系模板
2. M2 → M1: 模板匹配领域需求
   → 生成具体实体ID和关系声明
3. M1 → M0: 实体需求导出为事实采集清单
   → 输出"需要什么数据"的采集指令
4. M0 → 方案: 所有层填充完毕后
   → 执行推导引擎生成方案文档
```

**具体例子**（国转中心场景）：

```
M3: entity X : Organization { governance, mechanism }
    ↓
M2: entity ORG-国转中心 : Organization {
        governance: "理事会领导下的主任负责制"
        mechanism: "事业+企业双轨"
    }
    ↓
M1: TBox中加入国转中心的概念定义
    ABox中加入运行机制的断言
    ↓
M0: 产生数据采集需求：确认运行机制事实来源
    ↓
方案: 01-为什么必须做.md 中引用"双轨运行"作为论据
```

### 6.2 向上规约的实现

从M0到M3的模式抽象：

```
输入: M0 (具体事实) + M1 (当前模型) + 方案(文档)
输出: M2 (新规约规则) + M3 (语言扩展)

步骤:
1. M0 → M1: 数据聚类 → 发现重复模式
   → 抽象为实体类型或关系模式
2. M1 → M2: 模式形式化 → 编写规约规则
   → 加入到protocols/规约集
3. M2 → M3: 规约的元规则 → 如果规约类型不在现有元类型中
   → 扩展OntoLang语法(BNF扩展)
```

**具体例子**（从国转中心方案中抽象出的"断言追溯"规约）：

```
M0: 发现116个断言中101个没有SSOT编号引用
    ↓
M1: 识别为"断言可追溯性"问题模式
    ↓
M2: 编写P-SCHEME-002规约:
    "含'应该/必须/需要'的断言句至少70%有编号引用"
    ↓
M3: 将此规约类型形式化为OntoLang的protocol语法:
    protocol P-SCHEME-002 : AssertionTraceability {
        scope: "scheme/**/*.md"
        constraint: assertions_with_id / total_assertions >= 0.7
    }
```

### 6.3 向上与向下的协调

```
    向上规约(Induction)            向下推导(Deduction)
    ==================            ==================
    具体 → 抽象                   抽象 → 具体
    归纳 → 演绎                   类比 → 实例
    数据 → 模式                   规则 → 应用
    经验 → 规范                   理论 → 实践
    
    从方案提取规约                 从规约生成方案
    从实例抽象元模型               从元模型生成实例
    从实践沉淀形式语言             从形式语言指导实践
```

两者形成完整的知识螺旋(Knowledge Spiral)：

```
           M3
    向上 ↗    ↖ 向下
   M2 ←────────→ M2
    向上 ↗    ↖ 向下
   M1 ←────────→ M1
    向上 ↗    ↖ 向下
   M0 ←────────→ M0
     (事实)    (方案)
```

每一轮螺旋，系统的形式化程度提升一层，方案质量提升一层。
