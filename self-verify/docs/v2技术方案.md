# OntoDerive v2 渊衍框架技术方案

## 一、为什么需要v2：当前框架的六维诊断

> **本章要回答的问题**：为什么OntoDerive v1不够用？哪些理论缺口需要补充？(D-F1~D-F3)
>
> **是什么**：对v1框架的六维诊断——从贝叶斯、逻辑论、信息论、图灵机、系统论、控制论、模型理论七个视角分析v1的不足。(D-F1~D-F3)
> **为什么**：v1在国转中心方案(27个SSOT文件, 19条规约, 5条推理规则)的实战中被验证了基本可行性，但v1的置信度是离散的(4级, D-F4)、推理是硬编码的(Python, D-F6)、度量是缺失的(无熵, D-F8)、闭环是手动的(--rounds需人工介入, D-F10)。(D-F4~D-F10)
> **怎么做**：逐一从七个视角分析v1的限制，识别具体缺口。(D-F1~D-F3)

### 1.1 诊断方法(D-F1~D-F3)

将v1框架按七个理论视角逐一扫描，每个视角评估两个维度：**覆盖度**(v1当前能做什么)和**缺失度**(v1不能做什么)。

| 视角 | v1覆盖度 | v1缺失度 | 评分 |
|------|---------|---------|------|
| 贝叶斯 | 4级置信度标签(fact/inference/hypothesis/estimated) | 无连续概率、无信念传播、无贝叶斯更新 | 2/5 |
| 逻辑论 | `derives_from`自然语言标注, 5条硬编码推理规则 | 无形式蕴含(⊨)、无证明论、无一致性检测 | 3/5 |
| 信息论 | 无 | 无信息熵、无信息增益、无压缩比 | 0/5 |
| 图灵机 | --rounds N多轮迭代 | 无形式化状态机、无快照、无可重放 | 1/5 |
| 系统论 | derive→check→report循环 | 无自动感知、无自适应、无学习 | 2/5 |
| 控制论 | 规约检查(通过/不通过) | 无PID控制、无自适应阈值、无稳定性分析 | 1/5 |
| 模型理论 | 8元类型+4元关系(隐式MOF M2层) | 无显式M3/M2/M1/M0分层、无形式语言(BNF) | 2/5 |

**综合评分**: 11/35 = 31.4% (D-F1: 覆盖度总分/最高分=D-F2)

> **诊断结论**: v1在"有"和"无"之间存在大量断层。六个理论视角中信息论完全缺失(0/5)，图灵机和控制论接近空白(1/5)。v2的核心任务是填补这些断层。

### 1.2 七个视角的具体缺口(D-F4~F-F10)

**缺口1: 无概率量化 (贝叶斯)**
- 现状: `confidence: high/medium/low` 是离散的
- 问题: 当事实基座更新时，不能自动重新计算依赖推论的置信度
- 后果: 事实变了但推论置信度不变→知识库不一致(D-F4)
- 可证伪条件: 如果v2上线后置信度更新延迟>1次推导周期，则需要重新评估贝叶斯层架构(D-F5)

**缺口2: 无形式推理 (逻辑论)**
- 现状: 推理规则是Python硬编码的
- 问题: 无法进行形式化证明验证
- 后果: 推导链的正确性取决于代码质量，而非逻辑正确性(D-F6)
- 可证伪条件: 如果OntoLang解析器在100个测试用例中有>5个解析错误，则需要重新设计BNF语法(D-F7)

**缺口3: 无量化度量 (信息论)**
- 现状: 无信息熵度量
- 问题: 不知道知识库的不确定性
- 后果: 无法判断"添加一个新事实能减少多少不确定性"(D-F8)
- 可证伪条件: 如果KQI指数上线后与人工评估的相关性<0.7，则需要调整熵计算模型(D-F9)

**缺口4: 无计算模型 (图灵机)**
- 现状: --rounds N是简单循环
- 问题: 无状态持久化、无可重放
- 后果: 推导过程不可审计、不可回放(D-F10)
- 可证伪条件: 如果快照恢复时间超过推导时间的20%，则需要优化快照存储(D-F10)

## 二、v2融合架构：MOF四层 + 六层一体化

> **本章要回答的问题**：v2的整体架构是什么？各层之间的关系是什么？
>
> **是什么**：v2架构由MOF四层(M3/M2/M1/M0)和六层功能层(控制论/信息论/贝叶斯/逻辑/形式语言/本体核)组成。MOF四层定义"模型的层次"，六层定义"每一层的能力"。
> **为什么**：MOF四层确保架构的正交性(每个概念属于且只属于一个层次)，六层确保功能的完备性(七个理论视角都有对应的功能层)。
> **怎么做**：M3层定义OntoLang形式语言的语法，M2层实例化为10元类型+4元关系，M1层实例化为领域具体的TBox/ABox，M0层实例化为可验证的事实。

### 2.1 融合架构总览(D-F11~D-F12)

```
MOF四层                 六层能力
────────                ────────
M3: 元元模型层    ───   控制论层(PID+自演化)     → 系统自调节
                        信息论层(熵+知识质量指数)  → 量化度量
M2: 元模型层      ───   贝叶斯层(概率+信念传播)   → 不确定性推理
                        逻辑层(蕴含图+证明论)     → 形式推理
M1: 领域模型层    ───   形式语言层(OntoLang BNF)  → 精确表达
                        本体核(10元类型+元关系)    → 建模基础
M0: 实例/数据层   ───   计算基座(知识状态机)      → 计算执行
```

**设计原则(D-F13~D-F14)**:
1. **正交性**: MOF四层两两不相交(M3层概念不出现在M1层, D-F13)
2. **完备性**: 六层覆盖全部七个理论视角(控制论+信息论+贝叶斯+逻辑+形式语言+本体核=7个视角中的6个直接映射, 第7个图灵机映射到计算基座, D-F14)
3. **向上规约**: M0→M1→M2→M3 的模式抽象路径
4. **向下推导**: M3→M2→M1→M0 的实例化路径

### 2.2 正交性验证(D-F15)

| 层对 | 潜在交叉风险 | 隔离机制 |
|------|------------|---------|
| M3 ↔ M2 | M3定义语法, M2定义语义 | 语法≠语义, BNF产生式规则不引用具体类型 |
| M2 ↔ M1 | M2定义元类型, M1定义具体实体 | 元类型是类型构造子, 实体是类型实例 |
| M1 ↔ M0 | M1定义概念, M0定义数据 | 概念≠数据, TBox≠ABox |
| 控制论↔信息论 | 两者都涉及"度量" | 控制论度量偏差, 信息论度量不确定性 |

> 可证伪条件: 如果发现任一概念同时属于两个层次, 则正交性失效, 需重新分层(D-F15)

## 三、v2核心实体体系

> **本章要回答的问题**：v2框架涉及哪些实体？它们如何分类和标识？

> **是什么**：v2的实体体系——10种元类型, 8种关系, 18种ID前缀。
> **为什么**：统一的实体分类和ID体系确保知识的可追溯性。每个实体有唯一编号、明确类型、可验证的来源。
> **怎么做**：所有实体按元类型分类, ID采用标准前缀(D-F16~D-F18), 关系必须满足元关系矩阵(D-F16~D-F18)。

## 四、v2推理引擎设计

> **本章要回答的问题**：v2推理引擎如何工作？比v1多了哪些能力？
>
> **是什么**：v2推理引擎是一个融合了贝叶斯信念传播、逻辑蕴含检查、信息熵度量、PID反馈控制的知识状态机。
> **为什么**：v1引擎只有4条基本规约检查, 缺少概率推理、量化度量、自适应控制。v2将引擎从"检查工具"升级为"知识控制器"。
> **怎么做**：在v1的8条规约基础上增加贝叶斯层(置信度传播)、信息论层(熵计算)、控制论层(PID反馈)。

### 4.1 引擎架构(D-F19)

```
输入: 知识状态q_i(事实+实体+推论)
  │
  ├── [贝叶斯层] 信念传播 → 更新推论置信度
  ├── [逻辑层] 蕴含图检查 → 检测推导链断裂
  ├── [信息论层] 熵计算 → 评估知识库不确定性
  ├── [规约层] 8条规约检查 → 生成修复清单
  ├── [控制论层] PID反馈 → 生成自演化指令
  │
  └── 输出: 新知识状态q_i+1 + 修复清单 + 质量报告
```

### 4.2 规约扩展(D-F20)

v2在v1的8条规约基础上新增3条:

| 编号 | 名称 | 新增理由 | 覆盖的理论 |
|------|------|---------|-----------|
| C-09 | 信念传播检查 | v1无连续概率更新 | 贝叶斯 |
| C-10 | 知识熵阈值 | v1无不确定性度量 | 信息论 |
| C-11 | PID偏差跟踪 | v1无自适应阈值 | 控制论 |

## 五、实施路径

> **本章要回答的问题**：v2怎么从设计变成可运行的代码？
>
> **是什么**：v2的6个实施阶段(P0-P5)，每阶段的交付物和验收标准。
> **为什么**：阶段化实施确保每步可验证、可回退。
> **怎么做**：每阶段定义MoSCoW+验收标准+可证伪条件。

### 5.1 六阶段路线(D-F21~D-F24)

| Phase | 聚焦 | 核心交付 | 前置依赖 | 可证伪条件 |
|-------|------|---------|---------|-----------|
| P0 | 形式语言 | OntoLang BNF定义+解析器 | 无 | 解析器处理100个测试用例错误率<5% |
| P1 | 逻辑层 | 蕴含图+证明检查+一致性检测 | P0 | 逻辑检测覆盖>=90%的推导链(D-F22) |
| P2 | 贝叶斯层 | 概率本体+信念传播+置信度校准 | P1 | 置信度更新延迟<1次推导周期(D-F23) |
| P3 | 信息论层 | 知识质量指数+信息增益引导 | P1 | KQI与人工评估相关性>=0.7(D-F24) |
| P4 | 控制论层 | PID反馈+自适应阈值+自演化 | P2+P3 | 收敛速度相比v1提升>=50% |
| P5 | 图灵机 | 知识状态机+快照+可重放 | P0-P4 | 快照恢复时间<推导时间的20% |

### 5.2 资源估算(D-F25~D-F26)

| 资源 | 估算 | 依据 |
|------|------|------|
| 开发周期 | 12-16周(P0-P5顺序) | 每阶段2-3周 |
| 核心代码 | ~3000行(Python) | v1引擎约600行, v2为5倍复杂度 |
| 依赖 | Python 3.10+, lark(BNF解析) | OntoLang解析器需要LALR解析器 |
